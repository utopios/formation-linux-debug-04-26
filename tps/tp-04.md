# TP 4 — Analyser un goulot d'étranglement I/O

> Module : 6 — Disque, I/O et systèmes de fichiers
> Durée estimée : 1h
> Difficulté : 3 / 5
> Type : Travaux pratiques guidés

## Mise en situation

Une machine héberge deux services gourmands en écritures : `logger` et `archiver`. Le client signale que pendant les rotations nocturnes, les écritures applicatives partent « au ralenti ». Vous devez prouver quel service bloque le disque et sur quelle couche se joue le problème.

## Mise en place

Le formateur vous fournit deux scripts. Avant de commencer :

```bash
sudo bash setup-m6-tp04-logger.sh
sudo bash setup-m6-tp04-archiver.sh
```

Chaque script :
- Crée le source Python dans `/opt/<nom>/`
- Installe l'unité systemd correspondante
- Crée les répertoires de travail (`/var/log/logger/`, `/var/archive/`)

Aucun des deux services n'est pré-démarré : vous décidez du moment.

## Objectifs

- Caractériser un goulot d'I/O avec `iostat` et `iotop`
- Distinguer un problème côté filesystem d'un problème côté block device
- Proposer une piste d'atténuation testable

## Prérequis techniques

### Logiciels à installer

- `sysstat` (`iostat`, `pidstat -d`)
- `iotop`
- `fio` (pour la mini-charge de contrôle)

### Vérification de l'environnement

```bash
iostat -V
iotop --version
fio --version
```

## Architecture cible

```
VM Linux
  ├── /var/log/logger/           (écritures séquentielles, service logger)
  ├── /var/archive/              (écritures par gros chunks, service archiver)
  └── / (racine)                 (système)
```

## Étapes

### Étape 1 — Baseline (10 min)

Objectif : caractériser l'état du disque au repos.

1. Lancer un échantillonnage :
   ```bash
   iostat -xz 1 10 | tee iostat-baseline.log
   ```
2. Noter pour chaque device : `r/s`, `w/s`, `await`, `%util`, `aqu-sz`.
3. Capturer un snapshot filesystem :
   ```bash
   df -h
   df -i
   findmnt /var/log /var/archive
   ```

Point de contrôle : vous avez une référence pour pouvoir détecter la dégradation.

### Étape 2 — Reproduire la charge (25 min)

Objectif : reproduire le symptôme et l'observer.

1. Démarrer `logger` et `archiver` selon les instructions du formateur.
2. Dans un terminal :
   ```bash
   iostat -xz 1 60 | tee iostat-charge.log
   ```
3. En parallèle dans un autre terminal :
   ```bash
   sudo iotop -oPak | tee iotop-charge.log
   ```
   (taper `q` après 2 minutes pour sortir)
4. En parallèle encore :
   ```bash
   pidstat -d 1 60 | tee pidstat-charge.log
   ```

Point de contrôle : les trois logs sont sauvegardés, vous voyez au moins une phase de saturation.

### Étape 3 — Analyse (25 min)

Objectif : produire une conclusion argumentée.

1. Identifier le service le plus gourmand en I/O à partir des logs iotop et pidstat.
2. Dans `iostat-charge.log`, relever la période où `await` passe au-delà de 20 ms et `%util` proche de 100.
3. Pour la même période, comparer la profondeur de file `aqu-sz` à la baseline.
4. Décider si le problème est côté :
   - disque physique (latence haute sur IOPS modestes),
   - block layer (aqu-sz très élevée),
   - filesystem (beaucoup de petites écritures, sync, fsync),
   - ou applicatif (un seul process qui saturait déjà à lui seul la baseline).
5. Proposer une action d'atténuation testable (par ex. ajuster la politique de rotation, repousser l'archivage, séparer les volumes, changer l'option de mount `commit`).

Point de contrôle : une conclusion courte écrite, chiffrée par au moins deux métriques extraites des logs.

## Dépannage courant

<details>
<summary>Erreur : iotop refuse de démarrer (Permission denied)</summary>

Cause : iotop nécessite des capabilities spécifiques (CAP_NET_ADMIN, CAP_SYS_PTRACE).
Solution : lancer avec `sudo`, ou utiliser `pidstat -d 1` qui n'exige pas les mêmes droits.

</details>

<details>
<summary>Erreur : %util à 100% mais aucun service ne semble actif</summary>

Cause : écriture asynchrone depuis le cache page, flush par `kworker`/`flush-*`.
Solution : surveiller `vmstat 1` colonnes `bo` et `wa`, et vérifier `dirty_ratio` / `dirty_background_ratio` dans `/proc/sys/vm/`.

</details>

## Pour aller plus loin

Reproduire la charge avec une commande `fio` ciblée qui imite le profil du service dominant (séquentiel vs aléatoire, bs=4k vs bs=1M, queue depth). Comparer le verdict iostat avec et sans le service applicatif en parallèle, puis en déduire la marge restante.
