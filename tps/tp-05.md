# TP 5 — Étude de cas de dépannage de bout en bout

> Module : 7 — Tracing, réseau et méthodologie
> Durée estimée : 1h30
> Difficulté : 4 / 5
> Type : Travaux pratiques guidés

## Mise en situation

Un service `orderapi` expose une API HTTP. Ses p99 ont explosé dans la nuit. Les alertes remontent une latence anormale mais aucun outil de supervision ne pointe une cause claire. Vous avez accès SSH à la machine. Mission : trouver la cause racine en combinant les outils de la formation, selon la méthode USE.

## Mise en place

Le formateur vous fournit le script `setup-m7-tp05-orderapi.sh`. Avant de commencer :

```bash
sudo bash setup-m7-tp05-orderapi.sh
```

Le script :
- Crée `/opt/orderapi/orderapi.py` (l'API HTTP) et `/opt/orderapi/flock_holder.py` (déclencheur de latence)
- Installe deux unités systemd : `orderapi.service` et `orderapi-flock.service`

Démarrez **uniquement** `orderapi.service` au début. Vous (ou le formateur) déclencherez `orderapi-flock.service` pour reproduire l'incident pendant le TP.

## Objectifs

- Appliquer la méthode USE à un incident réel
- Combiner plusieurs outils (top, iostat, strace, perf, ss, bpftrace) pour converger
- Produire un post-mortem utilisable par un tiers

## Prérequis techniques

### Logiciels à installer

- Outils vus dans les modules précédents (procps, sysstat, strace, perf)
- `bpftrace` et `bpfcc-tools` (ou `bcc`)
- `ss`, `tcpdump`

### Vérification de l'environnement

```bash
bpftrace --version
ss -V
tcpdump --version
```

## Architecture cible

```
VM Linux
  ├── orderapi (HTTP sur port 8080)
  ├── dépendance : base PostgreSQL (localhost:5432)
  └── dépendance : cache Redis (localhost:6379)
```

## Étapes

### Étape 1 — Cadrage USE (15 min)

Objectif : poser les hypothèses sans ouvrir un seul outil avancé.

1. Sans toucher à la machine, lister les 4 ressources : CPU, mémoire, I/O, réseau.
2. Pour chacune, écrire :
   - une hypothèse à confirmer (U, S, E)
   - la commande qui testera cette hypothèse en moins de 30 secondes
3. Prioriser les hypothèses (la plus probable d'abord).

Point de contrôle : une liste écrite, signée par vous, avant tout diagnostic.

### Étape 2 — Diagnostic rapide (25 min)

Objectif : dérouler les hypothèses les plus probables.

1. Ressource CPU :
   ```bash
   uptime
   top -b -n 1 | head -20
   mpstat -P ALL 1 3
   ```
2. Ressource mémoire :
   ```bash
   free -h
   vmstat 1 5
   ```
3. Ressource I/O :
   ```bash
   iostat -xz 1 5
   sudo iotop -oPb -n 2
   ```
4. Ressource réseau :
   ```bash
   ss -s
   ss -tni
   ss -tn state close-wait
   ```

Point de contrôle : vous avez une colonne anormale identifiée. Si aucune, passer à l'étape 3.

### Étape 3 — Investigation ciblée (30 min)

Objectif : converger vers la cause racine.

Selon la piste identifiée, choisir une des trois branches ci-dessous (le formateur peut imposer la branche en fonction du scénario).

Branche A — latence interne (syscalls) :
```bash
sudo bpftrace -e '
  tracepoint:syscalls:sys_enter_read /comm == "orderapi"/ { @s[tid] = nsecs; }
  tracepoint:syscalls:sys_exit_read  /@s[tid]/
    { @ms = hist((nsecs - @s[tid])/1000000); delete(@s[tid]); }
'
```

Branche B — profil CPU :
```bash
perf record -F 99 -g -p $(pgrep orderapi) -- sleep 30
perf report --stdio | head -30
```

Branche C — dépendance réseau :
```bash
sudo tcpdump -i any -nn -c 500 port 5432 or port 6379 -w deps.pcap
# puis, une fois la capture arrêtée :
tcpdump -r deps.pcap -nn | head -40
```

Point de contrôle : une hypothèse confirmée ou infirmée par la commande choisie, et une piste suivante claire.

### Étape 4 — Post-mortem (20 min)

Objectif : livrer un compte-rendu exploitable.

1. Rédiger un post-mortem selon la trame :
   - Résumé (2 lignes)
   - Impact (utilisateurs, durée, sévérité)
   - Chronologie (horodatage des commandes)
   - Cause racine (preuve, pas intuition)
   - Action court terme
   - Action moyen terme
   - Détection : comment l'aurait-on vu plus tôt

Point de contrôle : le post-mortem est relisible par un collègue qui n'était pas dans le TP.

## Dépannage courant

<details>
<summary>Erreur : bpftrace retourne "ERROR: Could not resolve symbol"</summary>

Cause : noyau sans BTF (Linux kernel type information), ou tracepoint indisponible sur la version noyau utilisée.
Solution : vérifier `ls /sys/kernel/btf/vmlinux`, mettre à jour le noyau si possible, ou basculer sur une trace via kprobe équivalente.

</details>

<details>
<summary>Erreur : tcpdump perd des paquets (packets dropped by kernel)</summary>

Cause : buffer de capture trop petit, débit trop élevé.
Solution : augmenter le buffer avec `-B 4096` (KiB) ou filtrer plus finement.

</details>
