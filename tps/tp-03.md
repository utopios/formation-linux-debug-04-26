# TP 3 — Investiguer une application CPU-bound

> Module : 5 — Profiling avec perf
> Durée estimée : 1h30
> Difficulté : 4 / 5
> Type : Travaux pratiques guidés

## Mise en situation

Un binaire `hotcode` consomme un cœur à 100 % et traite les requêtes deux fois plus lentement que prévu. Vous devez trouver les fonctions responsables et proposer une piste d'optimisation, sans accès au code source dans un premier temps.

## Mise en place

Le formateur vous fournit le script `setup-m5-tp03-hotcode.sh`. Avant de commencer :

```bash
sudo bash setup-m5-tp03-hotcode.sh
```

Le script :
- Crée `/opt/hotcode/hotcode.c`
- Compile le binaire `/opt/hotcode/hotcode` avec les bonnes options pour perf (`-O2 -g -fno-omit-frame-pointer`)
- Clone FlameGraph dans `~/FlameGraph/`
- Baisse `kernel.perf_event_paranoid` à 1 pour permettre `perf` sans sudo

Considérez le binaire `/opt/hotcode/hotcode` comme une boîte noire pour la durée du TP : ne lisez pas le `.c` avant la fin.

## Objectifs

- Capturer un profil CPU propre avec perf
- Lire un perf report et naviguer dans le call graph
- Construire un flamegraph pour communiquer le résultat

## Prérequis techniques

### Logiciels à installer

- `linux-tools-common` et `linux-tools-generic` (pour `perf`)
- `git` (clone de FlameGraph)
- `perl` (exécution des scripts FlameGraph)

### Vérification de l'environnement

```bash
perf --version
perl -v
git --version
```

## Architecture cible

```
VM Linux
  ├── /opt/hotcode/hotcode        (binaire à profiler)
  └── ~/FlameGraph/               (scripts de génération)
```

## Étapes

### Étape 1 — Préparation (15 min)

Objectif : rendre perf utilisable et récupérer FlameGraph.

1. Autoriser perf en tant qu'utilisateur non root :
   ```bash
   echo 1 | sudo tee /proc/sys/kernel/perf_event_paranoid
   ```
2. Cloner FlameGraph :
   ```bash
   git clone https://github.com/brendangregg/FlameGraph.git ~/FlameGraph
   ```
3. Créer le dossier de travail :
   ```bash
   mkdir -p ~/tp3 && cd ~/tp3
   ```

Point de contrôle : `perf top` s'ouvre sans erreur, FlameGraph est cloné.

### Étape 2 — Première vue perf top (15 min)

Objectif : avoir une intuition avant de capturer.

1. Lancer `hotcode` selon les instructions du formateur, noter le PID.
2. Dans un autre terminal :
   ```bash
   perf top -p <pid>
   ```
3. Observer au moins une minute, noter les 3 premières lignes `Overhead + Symbol`.
4. Utiliser les touches `a` pour zoom sur un symbole, `Enter` pour ouvrir le call graph.

Point de contrôle : trois symboles identifiés et au moins un call graph sommaire.

### Étape 3 — Capture ciblée et rapport (30 min)

Objectif : produire un profil reproductible, analysable hors ligne.

1. Capturer 30 secondes avec call graph :
   ```bash
   perf record -F 99 -g -p <pid> -- sleep 30
   ```
2. Vérifier le fichier généré :
   ```bash
   ls -lh perf.data
   ```
3. Lire le rapport :
   ```bash
   perf report --stdio | head -40
   ```
4. Naviguer interactivement :
   ```bash
   perf report
   ```
5. Déplier la fonction dominante, remonter son call graph de 3 niveaux, noter la chaîne d'appels.

Point de contrôle : vous avez identifié une fonction responsable d'au moins 25 % du temps CPU et le chemin qui y mène.

### Étape 4 — Flamegraph et synthèse (30 min)

Objectif : produire une image partageable.

1. Convertir :
   ```bash
   perf script > out.stacks
   ~/FlameGraph/stackcollapse-perf.pl out.stacks > out.folded
   ~/FlameGraph/flamegraph.pl out.folded > hotcode.svg
   ```
2. Ouvrir `hotcode.svg` dans le navigateur.
3. Identifier la plus grande « plate-forme » au sommet des piles : c'est votre cible.
4. Proposer une hypothèse d'optimisation et justifier en une phrase.

Point de contrôle : un `hotcode.svg` lisible et une hypothèse d'optimisation formulée.

## Livrable attendu

Un fichier `rapport-tp3.md` et le fichier `hotcode.svg`. Le rapport contient :

- Les 3 symboles dominants de `perf report`
- La chaîne d'appels menant au plus gros consommateur
- Une phrase d'hypothèse d'optimisation et son impact estimé (ordre de grandeur)

## Dépannage courant

<details>
<summary>Erreur : perf event syscall returned with error EACCES</summary>

Cause : `perf_event_paranoid` trop restrictif.
Solution : baisser la valeur (`echo 1 | sudo tee /proc/sys/kernel/perf_event_paranoid`) ou lancer perf avec sudo.

</details>

<details>
<summary>Erreur : symboles affichés comme [unknown] ou 0x...</summary>

Cause : binaire strippé ou symboles noyau indisponibles.
Solution : installer les debug symbols du binaire (paquet `-dbg`) et `linux-image-<version>-dbgsym` pour le noyau, ou profiler avec `-F 49 -g dwarf` pour capturer via DWARF.

</details>

## Pour aller plus loin

Refaire la capture avec `-F 999` au lieu de `-F 99`. Comparer la taille du `perf.data`, le temps CPU consommé par `hotcode` pendant la capture, et la lisibilité du flamegraph. À quel moment la fréquence plus haute apporte vraiment plus d'information ?
