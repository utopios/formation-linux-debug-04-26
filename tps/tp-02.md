# TP 2 — Analyser une consommation mémoire anormale

> Module : 3 — Analyse mémoire
> Durée estimée : 1h
> Difficulté : 3 / 5
> Type : Travaux pratiques guidés

## Mise en situation

Un service `memhog` tourne sur votre VM. Son propriétaire affirme qu'il « tourne très bien en local » mais qu'en pré-production, la VM passe en swap puis le service est tué toutes les nuits. Vous devez confirmer ou infirmer la fuite et documenter la zone mémoire incriminée.

## Mise en place

Le formateur vous fournit le script `setup-m3-tp02-memhog.sh`. Avant de commencer :

```bash
sudo bash setup-m3-tp02-memhog.sh
```

Le script :
- Crée `/opt/memhog/memhog.py`
- Installe l'unité `memhog.service`
- Active un swap de 512 Mo (utile pour observer si/so avant l'OOM)

Aucun service n'est pré-démarré : c'est à vous de le faire au moment voulu.

## Objectifs

- Échantillonner la consommation mémoire d'un processus dans le temps
- Isoler la zone mémoire qui croît via `pmap`
- Relier l'OOM Killer au processus visé via `dmesg` et `/proc`

## Prérequis techniques

### Logiciels à installer

- `procps` (`free`, `vmstat`, `pmap`)
- `sysstat` (`pidstat`)
- Service `memhog` fourni par le formateur

### Vérification de l'environnement

```bash
free -h
vmstat 1 2
pidstat -V
```

## Architecture cible

```
VM Linux (2 vCPU, 2 Go RAM, 512 Mo swap)
  └── service memhog
```

## Étapes

### Étape 1 — Baseline (10 min)

Objectif : capturer l'état mémoire « à vide » avant démarrage du service.

1. Noter l'état :
   ```bash
   free -h
   ```
2. Vérifier qu'aucune pression mémoire n'est en cours :
   ```bash
   vmstat 1 5
   ```
3. Noter la valeur de `MemAvailable` en kB :
   ```bash
   grep MemAvailable /proc/meminfo
   ```

Point de contrôle : trois mesures cohérentes de la RAM disponible, swap à 0.

### Étape 2 — Observation sous charge (25 min)

Objectif : suivre la croissance mémoire du processus.

1. Démarrer le service selon la commande du formateur, puis récupérer le PID :
   ```bash
   pgrep -a memhog
   ```
2. Lancer un échantillonnage toutes les 5 secondes dans un fichier :
   ```bash
   while true; do
     date +%T
     grep -E '^(VmRSS|RssAnon|VmData|VmSwap)' /proc/<pid>/status
     echo "---"
     sleep 5
   done | tee memhog-sampling.log
   ```
3. Laisser tourner au moins 5 minutes.
4. En parallèle, prendre un premier `pmap` :
   ```bash
   sudo pmap -x <pid> > pmap-t0.txt
   ```
   Note : `sudo` est nécessaire si le service tourne en root (cas du service systemd ici), sinon `pmap` n'affiche qu'une seule ligne.
5. Au bout de 5 minutes, prendre un second pmap :
   ```bash
   sudo pmap -x <pid> > pmap-t5.txt
   ```
6. Faire un diff ciblé sur les 20 plus grosses zones :
   ```bash
   diff <(sort -k2 -n -r pmap-t0.txt | head -20) <(sort -k2 -n -r pmap-t5.txt | head -20)
   ```

Point de contrôle : `RssAnon` augmente de façon monotone, et `pmap` permet de nommer la zone qui grossit (anon, heap, mmap).

### Étape 3 — Confrontation à l'OOM Killer (25 min)

Objectif : confirmer le scénario de kill nocturne.

1. Continuer à laisser tourner `memhog` jusqu'à saturation de la RAM.
2. Dans un autre terminal, surveiller la pression :
   ```bash
   vmstat 1
   ```
3. Dès qu'un kill survient, récupérer la trace :
   ```bash
   sudo dmesg -T | grep -iE 'killed process|oom' | tail -20
   ```
4. Relever : PID, nom de process, score OOM, RAM consommée au moment du kill.
5. Si le service avait dû être préservé, noter la valeur d'`oom_score_adj` qui aurait été appropriée.

Point de contrôle : vous avez la ligne `dmesg` du kill et vous savez comment l'auriez-vous protégé (ou pas).

## Livrable attendu

Un fichier `rapport-tp2.md` incluant :

- Courbe `RssAnon` dans le temps (texte suffit : début, mi-parcours, avant-kill)
- Nom et taille de la zone mémoire qui a grossi entre T0 et T5
- Ligne dmesg du kill et interprétation
- Recommandation en une phrase pour le propriétaire du service

## Dépannage courant

<details>
<summary>Erreur : la VM freeze pendant la saturation</summary>

Cause : swap quasi plein, le noyau ralentit fortement toute la machine avant de tuer.
Solution : prévoir un accès console hors SSH (hyperviseur), ou baisser la taille limite de `memhog` pour forcer un kill plus rapide.

</details>

<details>
<summary>Erreur : dmesg n'affiche rien</summary>

Cause : le service a été arrêté par systemd (OOMScore, MemoryMax cgroup) avant que le noyau n'intervienne.
Solution : inspecter `systemctl status memhog.service` et `journalctl -u memhog.service` pour trouver le vrai motif de l'arrêt.

</details>

## Pour aller plus loin

Reconduire l'étape 2 avec `heaptrack` (ou `valgrind --tool=massif`) hors production pour obtenir un appel par appel la zone qui alloue. Comparer au diagnostic obtenu avec `pmap` seul : plus précis, mais à quel coût ?
