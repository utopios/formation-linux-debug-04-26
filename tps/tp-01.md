# TP 1 — Investiguer une application qui freeze et un scénario de crash

> Module : 2 — Debugger une application en cours d'exécution
> Durée estimée : 1h
> Difficulté : 3 / 5
> Type : Travaux pratiques guidés

## Mise en situation

Vous reprenez l'astreinte d'un collègue. Deux incidents sont ouverts. Le premier concerne un service `slowboot` qui ne termine pas son démarrage après 10 minutes. Le second concerne un binaire `cruncher` qui plante régulièrement. Vous devez investiguer les deux, sans code source, en vous appuyant uniquement sur les outils du module.

## Mise en place

Le formateur vous fournit deux scripts : `setup-m2-tp01-slowboot.sh` et `setup-m2-tp01-cruncher.sh`. Avant de commencer le TP, lancez-les sur votre VM :

```bash
sudo bash setup-m2-tp01-slowboot.sh
sudo bash setup-m2-tp01-cruncher.sh
```

Chaque script :
- Crée les sources C dans `/opt/<nom>/`
- Compile le binaire
- Pour `slowboot` : installe l'unité systemd `slowboot.service`
- Pour `cruncher` : configure les core dumps (`core_pattern`, `limits.conf`)

Aucun binaire n'est pré-installé : tout est généré par les scripts.

## Objectifs

- Diagnostiquer un processus bloqué et nommer le syscall responsable
- Capturer un core dump et en extraire une première information utile
- Produire un court compte rendu reproductible

## Prérequis techniques

### Logiciels à installer

- `strace` version 5 ou plus
- `gdb` version 10 ou plus
- `coreutils` standard
- `file`, `ldd`, `readelf`

### Vérification de l'environnement

```bash
strace --version
gdb --version
ulimit -c
```

## Architecture cible

```
laptop
  └── VM Linux (Ubuntu 22.04 ou équivalent)
        ├── /opt/slowboot/          (binaire + service systemd)
        └── /opt/cruncher/          (binaire fourni)
```

## Étapes

### Étape 1 — Préparer l'environnement (10 min)

Objectif : s'assurer que les core dumps sont autorisés et trouvables.

1. Vérifier la limite actuelle :
   ```bash
   ulimit -c
   ```
2. Activer les core dumps pour la session :
   ```bash
   ulimit -c unlimited
   ```
3. Localiser le pattern de core :
   ```bash
   cat /proc/sys/kernel/core_pattern
   ```
4. Créer un dossier de travail pour les traces :
   ```bash
   mkdir -p ~/tp-linux-debug && cd ~/tp-linux-debug
   ```

Point de contrôle : `ulimit -c` retourne `unlimited` et le dossier de travail existe.

### Étape 2 — Débloquer slowboot (25 min)

Objectif : identifier précisément ce que `slowboot` attend au démarrage.

1. Reproduire le freeze en lançant le service (commande fournie par le formateur : `sudo systemctl restart slowboot` ou lancement direct du binaire).
2. Retrouver le PID du processus :
   ```bash
   pgrep -a slowboot
   ```
3. Attacher strace sur tous les threads, en sortie fichier :
   ```bash
   sudo strace -f -o slowboot.trace -p <pid>
   ```
4. Observer la trace pendant au moins une minute, puis détacher (Ctrl-C).
5. Inspecter les dernières lignes de la trace et repérer le syscall qui n'a pas de retour.
6. Compléter avec `/proc/<pid>/wchan` et `/proc/<pid>/stack` (ce dernier nécessite root) pour confirmer.

Point de contrôle : vous pouvez nommer le syscall bloquant, ses arguments, et formuler une hypothèse sur la cause (fichier manquant, socket non joignable, DNS, lock, etc.).

### Étape 3 — Autopsie de cruncher (25 min)

Objectif : obtenir un core dump exploitable puis lire l'état au crash.

1. Lancer `cruncher` selon les instructions du formateur et attendre le crash.
2. Localiser le fichier core généré (répertoire courant ou chemin indiqué par `core_pattern`).
3. Vérifier que le core est bien associé au bon exécutable :
   ```bash
   file core.<pid>
   ```
4. Ouvrir le core avec gdb :
   ```bash
   gdb /opt/cruncher/cruncher core.<pid>
   ```
5. Dans gdb :
   ```
   bt
   info registers
   info threads
   ```
6. Relever le signal reçu (visible dans le bandeau initial de gdb), la fonction au sommet de la backtrace et le thread fautif.

Point de contrôle : signal identifié, dernière fonction nommée, thread fautif noté.

## Livrable attendu

Un fichier Markdown `rapport-tp1.md` contenant :

- Pour `slowboot` : PID observé, extrait de trace pertinent, conclusion en 3 à 5 lignes
- Pour `cruncher` : signal reçu, backtrace résumée, hypothèse sur la cause

## Dépannage courant

<details>
<summary>Erreur : Operation not permitted lors du strace</summary>

Cause : le process tourne sous un autre utilisateur (souvent root), ou `yama/ptrace_scope` bloque l'attachement non privilégié.
Solution : relancer la commande avec `sudo`, ou ajuster temporairement `echo 0 | sudo tee /proc/sys/kernel/yama/ptrace_scope` pour la session.

</details>

<details>
<summary>Erreur : aucun core file généré</summary>

Cause : `ulimit -c` à 0, ou `core_pattern` redirige vers un gestionnaire (apport, systemd-coredump).
Solution : vérifier `ulimit -c unlimited` dans le shell qui lance le binaire. Sous systemd-coredump, récupérer via `coredumpctl list` puis `coredumpctl gdb <PID>`.

</details>
